# Changelog

All notable changes to Mimir are documented in this file.

The format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/). Mimir follows [Semantic Versioning](https://semver.org/spec/v2.0.0.html) for published releases; version policy is documented in `PRINCIPLES.md`.

## [Unreleased]

### Added — launch publication assets

- **Launch article** — added the public launch article and linked it from the docs index and README.
- **Hidden planning archive** — moved historical planning notes into `.planning/planning` and removed stale public-doc links to planning scratch.

### Changed — public launch surface

- **Public artifact cleanup** — moved the recovery benchmark harness and fixtures into `benchmarks/recovery`, removed scratch prototype research fixtures from the tracked public tree, and rewrote stale public references to deleted private research artifacts.

### Added — harness context

- **Bounded context capsule** — `mimir context [--project-root <dir>] [--config <file>] [--limit <records>]` now renders a read-only, data-only context surface for agents and operators. It includes setup/readiness metadata, source-marked governed canonical records, truncation status, and pending-draft counts without printing raw draft text.
- **Operator memory controls** — `mimir memory list|show|explain|revoke` now gives operators a governed-log inspection surface. List/show/explain are read-only and render canonical records with the same data-only boundary as rehydration; revoke stages an append-only librarian review draft instead of deleting or mutating `canonical.log`.
- **Project doctor** — `mimir doctor` now gives first-run users one metadata-only project readiness command that prioritizes config, workspace identity, draft backlog, native Claude/Codex setup, librarian mode, remote sync, and capture-summary actions without printing raw draft text.
- **Pre-compaction hook validation** — `mimir hook-context` now validates the checkpoint inbox route before presenting it and reminds wrapped agents to checkpoint durable decisions before compaction or long handoff. Claude setup now installs and validates both `SessionStart` and `PreCompact` hook entries; Codex setup validates the supported `SessionStart` route.
- **Lightweight after-capture handoff** — `librarian.after_capture = "archive_raw"` now drains captured drafts without invoking an LLM by committing raw pending-verification evidence through the librarian path. After-capture mode remains a per-repo policy: first-run config templates and `mimir config init` default to rigorous `process`, while `archive_raw` is available for repos that prefer lightweight drainage.

### Fixed — harness isolation

- **Project-native setup doctor policy** — `mimir doctor` now treats completely missing project-local Claude/Codex setup as informational instead of action-required, so public repositories can forbid agent artifacts while still reaching a green readiness state. Partial native setup remains actionable.
- **Wrapped-session environment isolation** — `mimir status`, `mimir health`, `mimir drafts`, and `mimir remote` now ignore inherited wrapper `MIMIR_CONFIG_PATH` / `MIMIR_DRAFTS_DIR` when an explicit `--project-root` is supplied without `--config`, so inspecting sibling repositories cannot accidentally read the current wrapped session's Mimir state. Project-local config also wins over inherited `MIMIR_DRAFTS_DIR` for status, health, launch, drafts, and remote planning.
- **Binary smoke-test isolation** — the real `mimir rustc --version` subprocess smoke test now scrubs inherited `MIMIR_*` variables before launch, preventing local test runs from staging operator-live draft envelopes.
- **Quorum adapter spawn stability** — adapter and synthesis execution now retry transient Unix `ETXTBSY` spawn failures, closing a local parallel-test flake where freshly written executable shims could briefly report "text file busy".

### Added — recovery benchmark corpus

- **Structured recovery scenario data** — the worked recovery-benchmark scenario now has a machine-readable JSON companion with baselines, cold-start prompt, work-instruction test, typed ground truth, and staleness probes. A harness regression test validates scenario files so Category 8 can consume locked data instead of prose-only examples.
- **Recovery benchmark scenario schema validation** — scenario JSON now fails at load time when required baseline, ground-truth, or staleness-probe fields are malformed.
- **Recovery benchmark scenario ID validation** — scenario loading now rejects duplicate ground-truth or staleness-probe IDs before downstream scorecards can become ambiguous.
- **Recovery benchmark dry-run planner** — `./bench recovery --list` and `./bench recovery --scenario <id> --dry-run` now read structured scenario JSON and render deterministic baseline transcript / scorecard artifact plans without launching agents. A Python self-test covers the planner.
- **Recovery benchmark results initializer** — `./bench recovery --scenario <id> --init-results` now writes non-clobbering transcript placeholders, `scorecard.md`, `scores.json`, `notes.md`, and `run-plan.json` from scenario data. The harness self-test verifies creation and rerun safety.
- **Recovery benchmark environment scaffolds** — `./bench recovery --scenario <id> --prepare-envs` now writes non-clobbering per-baseline environment manifests, prompt files, preserved-state notes, materialized-input contracts, and setup README files under each scenario result directory.
- **Recovery benchmark environment validation** — `./bench recovery --scenario <id> --validate-envs` now validates per-baseline materialized-input contracts, treating baseline A as launch-ready by default and B/C/D as blocked until their preserved-state inputs are path-backed and verified.
- **Recovery benchmark input path validation** — environment validation now checks verified materialized inputs exist as the declared file/directory/manual kind and reports path/type failures separately from missing inputs.
- **Recovery benchmark launch contracts** — `./bench recovery --scenario <id> --launch-plan` now emits non-executing per-baseline launch contracts from validated environments, withholding contracts for blocked baselines.
- **Recovery benchmark materialized launch contracts** — `./bench recovery --scenario <id> --write-launch-contracts` now writes per-baseline `launch-contract.json` artifacts only when every baseline environment is ready.
- **Recovery benchmark launch contract validation** — `./bench recovery --scenario <id> --validate-launch-contracts` now checks materialized launch contracts for missing, invalid, or stale/tampered files before execution.
- **Recovery benchmark launch prompt validation** — launch-contract validation now rejects missing or tampered cold-start and work-instruction prompt files before execution.
- **Recovery benchmark transcript validation** — `./bench recovery --scenario <id> --validate-transcripts` now checks per-baseline transcript files for missing or placeholder captures before scoring.
- **Recovery benchmark transcript prompt validation** — transcript validation now requires captured files to include the scenario cold-start prompt and work-instruction test before scoring can proceed.
- **Recovery benchmark scoring evidence gate** — score and summary validation now require captured transcript evidence before marking a scenario complete or emitting scenario verdicts.
- **Recovery benchmark cutoff validation** — structured scoring now rejects `time_to_productive_minutes` values beyond the scenario cutoff and requires `DNR` for missed-cutoff recoveries.
- **Recovery benchmark staleness validation** — structured scoring now rejects stale-occurrence counts above the scenario's declared staleness probe count.
- **Recovery benchmark hallucination denominator validation** — structured scoring now rejects zero decision-assertion totals so a missing hallucination denominator cannot score as a perfect rate.
- **Recovery benchmark integer validation** — structured scoring now rejects JSON booleans in integer metric fields instead of silently treating them as `0` or `1`.
- **Recovery benchmark missing-score reporting** — aggregate summaries now include every scenario fixture and report missing `scores.json` files as incomplete evidence instead of silently skipping them.
- **Recovery benchmark score validation** — result initialization now includes a structured `scores.json` template, and `./bench recovery --scenario <id> --score-results` reports incomplete fields or emits per-baseline metric summaries from filled scores.
- **Recovery benchmark summary reporting** — `./bench recovery --summary-results` now scans scenario result directories and reports complete vs incomplete structured score files with baseline summaries for complete scenarios.
- **Recovery benchmark verdict aggregation** — summary reporting now applies the default threshold rubric, including D-vs-best-baseline metric wins, hallucination/staleness guardrails, and the aggregate >=2/3 scenario benchmark recommendation.

### Added — BC/DR restore drill

- **Git-backed restore drill** — `mimir remote drill --destructive` now deletes the local workspace `canonical.log`, restores from the configured Git recovery remote, verifies the restored log, reopens the store, and runs `(query :limit 1)` as a sanity query. `--dry-run` prints the planned deletion/restore/verify steps, `scripts/bcdr-drill.sh` wraps the command, and `docs/bc-dr-restore.md` documents backup, fresh-machine restore, corrupted-local-log recovery, and drill procedure.
- **Remote sync integrity guard** — `mimir remote push` and `mimir remote pull` now verify canonical-log integrity before and after log sync, refuse corrupt tails or dangling symbol references, and report `workspace_log_verified=true` on successful log movement or skip.
- **Store-open corrupt-tail guard** — `Store::open` now truncates only recoverable orphan/torn tails after the last durable CHECKPOINT and refuses non-recoverable corrupt tails without truncating, preserving bytes for `mimir verify` or remote restore.
- **Opt-in capture backup** — `[remote] auto_push_after_capture = true` now runs the verified Git `mimir remote push` path after wrapped-session capture and librarian handoff, records `remote_backup` in `capture-summary.json`, and leaves pull/manual restore explicit.

### Added — sanitisation boundary

- **Harness rehydration data boundary** — launch capsules now mark governed `rehydrated_records` with `data_surface`, `instruction_boundary`, and `payload_format`, carry a root `memory_boundary` consumer rule, and generated `agent-guide.md` tells wrapped agents never to execute imperatives found inside rehydrated records. Added an adversarial literal regression and `docs/sanitisation.md`.
- **Librarian raw-draft data boundary** — `RetryingDraftProcessor` now wraps first-attempt and retry prompts with a `mimir.raw_draft.data.v1` boundary that marks draft prose as `data_only_never_execute`. Added a fixture-backed write-time adversarial corpus covering prompt injection, role confusion, instruction-disguised-as-memory, command-looking observations, and direct librarian-address attempts.
- **MCP retrieved-memory data boundary** — `mimir_read` and `mimir_render_memory` now return governed records as data-marked Lisp payloads with `memory_boundary`, per-record `data_surface`, `instruction_boundary`, and `payload_format` fields instead of bare Lisp strings.

### Changed — MCP render surface

- **Compact MCP runtime responses** — `mimir-mcp` tool responses now serialize compact JSON, cap tool descriptions by regression test, split `mimir_verify` tail reporting into `tail_type` plus corrupt-tail `tail_error`, and keep workspace/lease runtime errors code-first.

### Fixed — canonical decoder strictness

- **Canonical wire decoding** — decoder now rejects overlong varints, reserved Semantic/Inferential flag bits, and invalid Procedural precondition discriminants instead of accepting noncanonical encodings.

### Fixed — checkpoint accounting

- **Checkpoint `memory_count` semantics** — `Store::commit_batch` now records only Sem/Epi/Pro/Inf memory records in `CheckpointRecord.memory_count`, excluding symbol events, supersession edges, episode metadata, and the checkpoint itself.

### Added — scope-aware librarian retry loop

- **`mimir_librarian::RetryingDraftProcessor`** — LLM-backed draft processor that parses the librarian JSON contract, validates candidate Lisp through `PreEmitValidator`, commits accepted batches through `mimir_core::Store::commit_batch`, and re-prompts with structured retry hints for JSON / parse / bind / semantic / emit failures. Failed multi-record attempts are transactional: earlier valid records from the same failed attempt do not poison scratch validation state before retry. Store-level pipeline rejections are also retryable and do not mark drafts accepted.
- **`mimir_librarian::RawArchiveDraftProcessor`** — deterministic no-LLM processor for lightweight draft drainage. `mimir-librarian run --archive-raw` commits each raw draft as low-confidence `pending_verification` evidence plus librarian-assigned provenance/data-boundary facts through the normal workspace lock and append-only store, then leaves later LLM consolidation optional.
- **`mimir-librarian run`** — now uses the retrying processor by default. `--defer` preserves the lifecycle-only dry run, `--max-retries` controls the retry budget, and `--llm-timeout-secs` controls the Claude subprocess timeout.
- **Supersession-conflict policy** — deterministic equal-key supersession conflicts now skip by default instead of asking the LLM to guess a repaired timeline. `--review-conflicts` writes a structured JSON review artifact under `drafts/conflicts/` and quarantines the draft.
- **Exact duplicate filtering** — before commit, the retrying processor compares candidate Semantic, Episodic, Procedural, and Inferential records against the replayed canonical store and skips exact duplicates. Mixed batches keep only unique records, so duplicates do not force a supersession-conflict review.
- **Episodic retention in core** — `mimir_core::Pipeline` now retains and replays committed `EpiRecord`s via `Pipeline::episodic_records()`, enabling exact duplicate checks and future event-oriented reads.
- **Configurable valid-at dedup window** — `mimir_librarian::DedupPolicy` and `LibrarianConfig::dedup_valid_at_window` let the retrying processor skip otherwise-identical Semantic and Inferential candidates whose `valid_at` clocks are inside the default one-day window. `--dedup-valid-at-window-secs 0` restores exact-only behavior.
- **Inferential librarian responses** — the LLM response contract now accepts `"kind":"inf"` candidates, so Inferential duplicate filtering and durable commit can be exercised through the same processor path as the other memory types.
- **Consensus quorum draft intake** — `DraftSourceSurface` now includes `consensus_quorum`, `mimir-librarian submit --source-surface consensus-quorum` stages quorum outputs as governed evidence drafts with `source_agent = quorum` and episode provenance, and sweep rejects that surface because quorum results must preserve explicit episode provenance rather than ambient file discovery.
- **Consensus quorum envelopes** — added typed `QuorumEpisode`, `QuorumParticipantOutput`, `QuorumResult`, participant/vote/status enums, and `QuorumStore` file-backed episode create/load, participant output append/load, visibility-gated round reads, result save/load storage, and CLI synthesis adapter planning/running/acceptance. Model synthesis emits proposed result artifacts and still requires explicit acceptance plus the normal draft/librarian path.
- **Consensus quorum CLI artifacts** — `mimir-librarian quorum create|append-output|outputs|visible` records file-backed quorum episodes, appends participant outputs, loads round outputs, and exposes only store-approved visible prior outputs. This is a recorded-artifact contract for future adapters, not adapter execution.
- **Consensus quorum adapter requests** — added typed `QuorumAdapterRequest`, `QuorumStore::build_adapter_request`, and `mimir-librarian quorum adapter-request` so future Claude/Codex adapters can consume a stable JSON request with participant identity, question, evidence policy, and visibility-gated prior outputs.
- **Consensus quorum result synthesis command** — `mimir-librarian quorum synthesize` saves an explicit structured `QuorumResult` from supplied recommendation/status/confidence/votes/dissent fields and attaches stored participant output references as evidence. It records synthesis artifacts only; it does not run a model synthesizer.
- **Consensus quorum proposed draft emission** — `mimir-librarian quorum submit-drafts` loads a saved quorum result and submits each proposed memory draft into the normal pending draft store with `source_surface = consensus_quorum`, `source_agent = quorum`, episode provenance, project/operator metadata, decision-status tags, and consensus-level tags.
- **Consensus quorum recorded fixture smoke test** — binary coverage now exercises `quorum create -> adapter-request -> append-output -> synthesize -> submit-drafts` with recorded Claude/Codex fixture responses, proving the full recorded-artifact path reaches a pending `consensus_quorum` draft without live adapters.
- **Consensus quorum adapter plans** — `mimir-librarian quorum adapter-plan` now materializes native Claude and Codex command contracts from a stored adapter request, writes request/prompt/response artifact paths, and returns the matching `append-output --prompt-file --response-file` command so participant responses still enter through the audited quorum store rather than direct memory writes.
- **Consensus quorum adapter execution status** — `mimir-librarian quorum adapter-run` executes a Claude/Codex adapter plan with a bounded timeout, redirects stdin/stdout through prompt/response artifacts, writes a structured status JSON with byte counts and the matching append command, and deliberately does not append participant outputs itself.
- **Consensus quorum round adapter execution** — `mimir-librarian quorum adapter-run-round` executes every participant adapter for one deliberation round, writes an aggregate round status artifact, preserves independent-first visibility gates, and still leaves participant-output recording to explicit `append-output` commands.
- **Consensus quorum status append** — `mimir-librarian quorum append-status-output` validates successful adapter-run or adapter-run-round status artifacts, checks the prompt/response artifacts still match the recorded metadata, and records participant outputs through the existing `append-output` command path.
- **Consensus quorum multi-round adapter driver** — `mimir-librarian quorum adapter-run-rounds` sequences independent, critique, and revision rounds through `adapter-run-round` status artifacts and `append-status-output` gates, so later rounds only see prior outputs after they have been recorded through the audited append path.
- **Consensus quorum synthesis hardening** — `mimir-librarian quorum synthesize-run` status now separates native process status from proposed-result validity with `process_status`, `result_valid`, and `validation_error` fields. `accept-synthesis` accepts either `--result-file` or a successful `--status-file`, validates status/result artifact consistency, and direct `synthesize` now validates proposed result text fields, confidence values, duplicate votes, and exactly one participant vote per episode participant before saving a `QuorumResult`.
- **Consensus quorum pilot plans** — `mimir-librarian quorum pilot-plan` writes a replayable manifest for a live Claude/Codex quorum pilot, including exact command argv for multi-round adapter execution, synthesis, status-backed acceptance, and draft submission plus the expected status/result artifact paths. The manifest is planning-only and preserves the existing append, accept, and librarian draft gates.
- **Consensus quorum pilot status** — `mimir-librarian quorum pilot-status --manifest-file PATH` reads a pilot manifest and reports each gate as pending, complete, or failed by checking round status artifacts, appended participant outputs, synthesis status/result validation, accepted `result.json`, and submitted `consensus_quorum` drafts.
- **Consensus quorum pilot run** — `mimir-librarian quorum pilot-run --manifest-file PATH` executes a pilot manifest by replaying the recorded `adapter-run-rounds`, `synthesize-run`, `accept-synthesis`, and `submit-drafts` commands in order, then returns the final `pilot-status` result. It now reports failed steps with partial run state, returns a nonzero CLI exit for unsuccessful runs, and skips gates that are already complete on rerun. It uses the existing gates rather than adding a direct-write shortcut.
- **Consensus quorum pilot review** — `mimir-librarian quorum pilot-review --manifest-file PATH --reviewer ID --decision pass|needs-work|fail --summary TEXT` writes a non-participant certification artifact with findings, next actions, and a full status snapshot. `pass` requires complete `pilot-status`; needs-work/fail can record incomplete pilots without certifying them.
- **Consensus quorum pilot summary** — `mimir-librarian quorum pilot-summary --manifest-file PATH` renders the operator-facing pilot snapshot: gate status, accepted result presence, proposed-memory-draft count, submitted `consensus_quorum` draft count, review presence/decision, and next action.
- **Consensus quorum required draft exit criteria** — `pilot-plan --require-proposed-drafts N` records a minimum proposed-memory-draft requirement in the manifest. `pilot-status` and `pilot-run` now fail the acceptance/submission gates when the accepted result proposes fewer drafts than required, preventing a review pass from certifying a live pilot that never exercised draft submission.
- **Live quorum pilots** — ran real local Claude/Codex independent-round pilots from manifests. The first sandboxed attempt failed at `run_rounds` and produced structured failure status; the escalated rerun completed participant execution, synthesis, status-backed acceptance, and review but proposed no memory drafts. A required-draft rerun with `--require-proposed-drafts 1` completed participant execution, synthesis, explicit acceptance, governed `consensus_quorum` draft submission, and pass review.
- **Service remote adapter boundary** — `mimir remote push --dry-run` and `mimir remote pull --dry-run` now succeed for `remote.kind = "service"` without network I/O, rendering a stable service-adapter contract with workspace identity, local log/draft status, append-only log prefix requirements, copy-only draft sync requirements, and the librarian-governed write boundary. Real service push/pull still fails until a transport adapter lands.

### Added — Inferential resolver wired (Phase 3.1 of the prime-time roadmap)

`:kind inf` queries now return authoritative Inferentials; re-derivation with the same `(s, p)` and a later `valid_at` auto-supersedes the prior Inferential at emit time. Closes the last memory-type gap in the read surface.

- **`mimir_core::pipeline::Pipeline`** — added `inferential_records: Vec<InfRecord>` and `inferential_by_sp_history: BTreeMap<(SymbolId, SymbolId), Vec<usize>>` (mirroring `semantic_records` / `semantic_by_sp_history`). Accessors `Pipeline::inferential_records()` and `Pipeline::inferential_history_at(s, p)` expose them for the resolver. Populated on both emit (commit path) and replay (log reopen). Clone-on-write preserves batch atomicity.
- **`SupersessionIndex.inferential_by_sp: BTreeMap<(SymbolId, SymbolId), CurrentSemantic>`** — current-state index for Inferential supersession detection, reusing the `CurrentSemantic` `(memory_id, valid_at)` shape since the two indices carry identical data.
- **`resolve_inferential_supersession`** (pipeline emit helper) — mirrors `resolve_semantic_supersession`: forward (later `valid_at` closes the old record's validity via a `Supersedes` edge + updated current-state entry), retroactive (earlier `valid_at` closes the *new* record's validity at the existing record's `valid_at`), conflict (equal `valid_at` → new `EmitError::InferentialSupersessionConflict`).

### Added — `mimir-mcp` write tools + workspace lease (Phase 2.3 of the prime-time roadmap)
- **`mimir_open_workspace`** — opens (or creates with the MIMR magic header) a canonical log at the supplied `log_path` and mints a 30-minute write lease. Returns `{workspace_id, log_path, lease_token, lease_expires_at}` where `lease_token` is a 32-char lowercase-hex 128-bit token derived from per-call `RandomState` instances. `ttl_seconds` argument allows per-call override (must be > 0, capped at 24h via `MAX_LEASE_TTL_SECONDS`); the server's default is taken from `MIMIR_MCP_LEASE_TTL_SECONDS` env var at startup, falling back to `DEFAULT_LEASE_TTL_SECONDS` (1800 = 30min). Errors with `lease_held` (and the existing lease's expiry) if a non-expired lease is already alive.
- **`mimir_write`** — wraps `mimir_core::store::Store::commit_batch`. Takes `{batch, lease_token}`; validates the lease via constant-time token comparison + non-expiry check + workspace-path sanity check, then dispatches the actual commit to `tokio::task::spawn_blocking` (filesystem + CPU work; would otherwise stall the reactor). Returns `{episode_id, committed_at}` with the auto-generated `__ep_N` symbol resolved against the symbol table. Error set: `no_workspace_open`, `no_lease`, `lease_expired`, `lease_token_mismatch`, `lease_workspace_mismatch`, `commit_failed: <reason>`.
- **`mimir_close_episode`** — convenience wrapper that emits `(episode :close)` as a write batch. Same lease + error contract as `mimir_write`. Communicates intent more clearly to agent runtimes than embedding the form in a free-form `mimir_write` call. (No `label` argument: the Mimir grammar does not currently accept keyword args on `(episode :close)` — labels and parents are set on `(episode :start …)`. If post-close labelling is needed, file a spec follow-up.)
- **`mimir_release_workspace`** — drops the lease. The store stays open so reads continue to work; subsequent writes require a fresh `mimir_open_workspace`. Errors with `no_lease` if no lease is held, `lease_token_mismatch` if the supplied token doesn't match (prevents accidental cross-caller release).
- **`mimir_status` extended** with `lease_held: bool` and `lease_expires_at: Option<String>` fields. Status is now `async` (it locks the new `Arc<Mutex<...>>` state slots).

### Changed — `MimirServer` state model (Phase 2.3)
- **`MimirServer.workspace_id`, `log_path`, and `store` are now wrapped in `Arc<Mutex<…>>`** so `mimir_open_workspace` can swap them at runtime. Previously they were set once at construction. The constructor signature is unchanged: callers pass `Option<WorkspaceId>`, `Option<PathBuf>`, `Option<Store>` and the wrapping is transparent. The `MimirServer: Clone` invariant is preserved (Arc-wrapped state is cheap to clone for the `#[tool_router]` macro's per-call dispatch).
- **New `MimirServer.lease: Arc<Mutex<Option<LeaseState>>>`** holds the in-memory single-writer lease state (token, expiry, workspace path). Per-server-instance, not persistent across restarts — fine for stdio MCP because the server is one-shot per session. Streamable-HTTP transport deployments may need persistence; tracked as a Phase 6 follow-up.
- **Lease expiry is lazy.** Every write tool re-checks `lease.expires_at` on entry; there is no background reaper. Expired leases only matter at the next write attempt; reads are unaffected. Documented in the `crate::server` module-level comment as the explicit design choice (vs. a per-second tokio::interval expiry sweep, which would add background-task complexity for negligible UX benefit).
- **Token validation uses constant-time comparison** (`constant_time_eq`). Stdio has no observable timing channel so this is currently free hardening, but it makes the code transport-agnostic for the eventual streamable-HTTP server.

### Added — `mimir-mcp` read tools (Phase 2.2 of the prime-time roadmap)
- **`mimir_read`** — wraps `mimir_core::pipeline::Pipeline::execute_query`. Takes a single Lisp `(query …)` string with the full predicate set documented in `read-protocol.md` (`:s`, `:p`, `:kind`, `:as_of`, `:as_committed`, `:limit`, `:include_retired`, `:include_projected`, `:confidence_threshold`, `:explain_filtered`, `:show_framing`, `:debug_mode`). Returns matched records rendered as Lisp via `LispRenderer::render_memory` (round-trippable through `mimir_core::parse`), the read-protocol flag bitset surfaced as lowercase strings (`stale_symbol`, `low_confidence`, `projected_present`, `truncated`, `explain_filtered_active`), and the snapshot-isolation watermarks (`as_of` / `as_committed` / `query_committed_at` as ISO-8601). Sync `Pipeline::execute_query` runs under `tokio::task::spawn_blocking` so the reactor stays free for concurrent tool calls.
- **`mimir_verify`** — wraps `mimir_cli::verify`. Read-only integrity check on a canonical log: decodes every record, counts checkpoints / memory records / symbol events, classifies trailing bytes (`clean` / `orphan_tail` / `corrupt: <decoder error>`), reports dangling symbol references. Defaults to the server's configured workspace log; accepts an optional `log_path` argument to verify a different file (useful for backups, forensics, copied logs).
- **`mimir_list_episodes`** — paginated list of registered episodes ordered by `committed_at` ascending (durability order). Each row carries the episode symbol name (e.g. `__ep_42`), ISO-8601 commit time, and parent episode id if registered. `limit` defaults to 50 and is capped at 1000; `offset` defaults to 0. Resolves `Pipeline::iter_episodes()` against the symbol table to produce caller-friendly names. Pagination is computed in a `spawn_blocking` block since the underlying `HashMap` iteration + sort can be O(n) on large session histories.
- **`mimir_render_memory`** — wraps `LispRenderer::render_memory` against the first record matched by a `(query …)` form. Returns the empty string for zero matches; errors with `multiple_matches` for >1 match (the contract is single-record by design — use `mimir_read` for multi-record rendering). Useful for "show me one specific memory" inspection where the caller knows the predicate uniquely identifies the target.
- **Workspace-required gating.** All four read tools error with `McpError::invalid_request("no_workspace_open: …", None)` when `MimirServer` was constructed without a `Store`. The binary opens a Store at startup if `MIMIR_WORKSPACE_PATH` is set; tests construct the server with `Some(Store)` directly. Phase 2.3's `mimir_open_workspace` tool replaces the env-var discovery with explicit per-session opening + write lease.

### Fixed — security latents (Phase 1.4 of the prime-time roadmap)
- **Decoder OOM on attacker-controlled count (Security F2 / P2)** — `decode_epi_body` (Epi participants), `decode_inf_body` (Inf `derived_from`), and `decode_episode_meta_body` (EpisodeMeta `retracts`) all called `Vec::with_capacity(count)` on a `count` read straight from a varint *before* the decode loop ran. An attacker who set `count` near `usize::MAX` (10-byte varint) triggered a multi-exabyte allocation that aborted the process via the global allocation handler — defeating every reader of `mimir-cli verify` / `decode` / `log` / `symbols` and the `Store::open` recovery path. Fix: cap each `with_capacity` by `bytes.len() - *offset` (a sound upper bound, since each decoded element consumes ≥1 byte). Three regression tests (`decode_{epi,inf,episode_meta}_does_not_oom_on_huge_*_count`) craft frames with `count = u64::MAX` and assert the decoder returns a typed `Err` instead of aborting.
- **Parser stack overflow on deep nesting (Security F3 / P2)** — `parse_value` → `parse_value_list_body` → `parse_value` (and `parse_correct` → `parse_form` → `parse_correct`) recurred without bound. An input like `(((…)))` of pathological depth blew the host stack uncatchably (no `Result::Err`, no `ParseError` variant — just process death). Fix: `Parser::depth` field tracked on every recursive entry, capped at the new public `MAX_NESTING_DEPTH = 256` constant, surfaces as the new `ParseError::NestingTooDeep { pos, max }` variant. 256 levels is generous by orders of magnitude (Mimir's grammar realistically nests 2–3 deep) and keeps worst-case stack consumption well under 1 MiB. Four regression tests cover the limit boundary, the over-limit case, the pathologically-deep case, and the `correct`-form recursion path.

### Changed — wire format (schema break)
- **Canonical log carries an 8-byte header** — `MIMR` ASCII magic + `u32` LE format version (currently `1`) — written eagerly when `CanonicalLog::open` is called against an empty file, validated on every reopen. Pre-fix logs (no header) fail to open with the new `LogError::IncompatibleFormat`; the open call surfaces the error **before** any truncation, append, or recovery logic, closing the destructive-truncate footgun where `Store::open` against a misrouted path (e.g., a config file, an arbitrary user file) silently zeroed the target. This is the v1.1 fresh assessment's Security F1 (P1) finding. Header is physically transparent: `LogBackend::{len, read_all, last_checkpoint_end, truncate}` all operate in logical bytes (record stream only) — `CanonicalLog` strips the header on the way out and adds `LOG_HEADER_SIZE` on the way in. In-memory test backends (`FaultyLog`) carry no header. **Any on-disk log written before this commit must be re-created**; the codebase is pre-1.0 and consumes only its own logs, so the migration cost is zero.

### Changed
- **Wire format (schema break)** — `MemoryFlags { projected, stale }` split into kind-specific `SemFlags { projected }` (1 byte, bit 0) and `InfFlags { projected, stale }` (1 byte, bits 0–1) in `mimir_core::canonical` (issue #32). `EpiRecord` and `ProRecord` no longer carry a `flags` field at all — their frame body is one byte shorter. `ir-canonical-form.md` § 5 updated; new § 5.5 documents per-kind flag layout and rationale (stale is only meaningful for Inferentials; shared byte pushed invariant enforcement out of the type system). Decoder enforces "body fully consumed" (`DecodeError::BodyUnderflow`), so pre-break logs with a trailing Epi/Pro flags byte hard-fail rather than silently losing a byte — new test `legacy_{epi,pro}_with_trailing_flags_byte_rejected` locks this in. Any on-disk log written before this commit must be re-encoded before replay.
